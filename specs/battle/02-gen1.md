# Pokémon Generation 1 Battle Mechanics Specification

> Types and interfaces reference `@pokemon-lib-ts/core` — see `core/01-entities.md` for canonical definitions.

## 1. Overview

Generation 1 (Red/Blue/Yellow) introduced the foundational battle system that would persist through later generations, but with several critical differences that make Gen 1 mechanically distinct:

- **No Abilities**: Pokémon have no abilities whatsoever. This drastically simplifies battle mechanics.
- **No Held Items**: Items exist in the game but cannot be held during battle.
- **Unified Special Stat**: A single "Special" stat governs both Special Attack and Special Defense. Later gens split this into SpA and SpD.
- **No Physical/Special Split Per Move**: The categorization of moves as physical or special is entirely type-based. All normal-type moves are physical, all grass-type moves are special, etc.
- **Type-Based Damage Scaling**: Physical moves use Attack/Defense, Special moves use Special/Special.
- **Simplified Status System**: No permanent stat modifications, simpler condition mechanics.
- **Critical Hit Based on Speed Stat**: Crit probability is determined by base Speed, not by held items or move properties.
- **No Priority System**: Except for Counter and Quick Attack, all moves resolve in Speed order. No priorities like +1, +2, etc.
- **No Weather, Terrain, or Hazards**: The battle environment is static.
- **No Stat Stages Interaction with Moves**: Some moves don't respect stat changes at all; others do. The interaction is buggy.
- **Multiple Critical Bugs**: The 1/256 miss bug, Ghost immunity bug, Focus Energy bug, and badge boost glitch are all present.

This is a fundamentally different system from Gen 3+ and requires its own complete implementation.

## 2. Stat System

Gen 1 uses the original stat formula with some quirks:

### Stat Formula

```typescript
interface Gen1Stats {
  hp: number;
  attack: number;
  defense: number;
  special: number;  // Unified Special stat (later split into SpA/SpD)
  speed: number;
}

function calculateGen1Stat(
  statValue: number,
  baseValue: number,
  iv: number,  // Individual Value: 0-15
  ev: number   // Effort Value: 0-65535
): number {
  // IV and EV divisions are integer division (floor)
  return Math.floor((2 * baseValue + iv + Math.floor(ev / 4)) * level / 100) + 5;
}

function calculateGen1HP(
  baseHP: number,
  iv: number,
  ev: number,
  level: number
): number {
  // HP has a special formula
  if (baseHP === 1) {
    return 1; // Ditto's special case
  }
  return Math.floor((2 * baseHP + iv + Math.floor(ev / 4)) * level / 100) + level + 5;
}
```

### Key Differences from Later Generations

1. **IVs are 4 bits** (0-15) instead of later gens' 5-6 bits. This means lower stat variance.
2. **EVs are applied in 4-point increments**: Each EV point increases a stat by 1 for every 4 EV invested at level 100.
3. **No stat softcap**: EVs can exceed 252 per stat and even exceed 65535 total. This was later capped at 252 per stat and 510 total.
4. **Special stat ranges from 1-317** at level 100 with maximum base/IV/EV.
5. **Speed stat is crucial**: Base 100+ Speed Pokémon have significantly higher crit rates, and Speed determines move order absolutely.

### Special as Unified Stat

The "Special" stat affects both:
- **Damage Output**: Special moves use Special as their attack stat
- **Damage Mitigation**: Special moves use Special as the defense stat
- **Critical Hit Rate**: Indirectly (not directly; Speed determines crit rate)

This means a Pokémon with low Special takes more damage from special moves AND deals less Special damage. A Pokémon with high Special both tanks and deals special moves effectively.

### Critical Bug: Ghost vs Psychic

**Bug Description**: In Gen 1, Ghost-type moves should be super-effective against Psychic types. However, due to a programming error, the damage multiplier for "Ghost super-effective against Psychic" was never applied in the damage formula. Instead, Ghost moves do normal damage (1.0x) to Psychic types.

**Effect**: Psychic types, which are already weak to Ghost thematically, are actually immune to Ghost-type moves in practice during Gen 1.

### Other Type Notes

- **Poison super-effective vs Bug**: Unlike later gens, Poison is super-effective against Bug in Gen 1.
- **Bug weak to Poison**: Correspondingly, Bug is weak to Poison.
- **No Steel type interactions**: Steel is Gen 2+.
- **Type immunity via Ghost and Normal**: Ghost moves do not affect Normal types (immunity, not resistance). Normal moves do not affect Ghost types (immunity).

## 4. Damage Formula

The Gen 1 damage formula is fundamentally different from Gen 3+ formulas. It uses integer division at multiple points, which creates specific damage rolls and thresholds.

### Complete Gen 1 Damage Formula

```typescript
interface DamageCalculationInputs {
  attack: number;           // Attack or Special stat of attacker
  defense: number;          // Defense or Special stat of defender
  basePower: number;        // Base Power of the move
  level: number;            // Level of attacking Pokémon
  isPhysical: boolean;      // true if using Attack/Defense, false if using Special
  typeMultiplier: number;   // 0.5, 1.0, 2.0, or 2.5 (Weakness/Resistance, see below)
  isCritical: boolean;      // true if move lands as critical hit
  randomFactor: number;     // 0.85 to 1.00 (multiplied in, not added)
  weatherEffect?: number;   // Rain (1.5x Water, 0.5x Fire) or Sun, if applicable
}

function calculateGen1Damage(inputs: DamageCalculationInputs): number {
  const {
    attack,
    defense,
    basePower,
    level,
    typeMultiplier,
    isCritical,
    randomFactor,
  } = inputs;

  // Step 1: Calculate base damage
  // (Level * 2 / 5 + 2) * Power * Attack / Defense / 50 + 2
  let damage = Math.floor(
    Math.floor(
      Math.floor(
        (level * 2) / 5 + 2
      ) * basePower * attack
    ) / defense
  ) / 50 + 2;

  // Step 2: Apply critical hit (ignores stat stages)
  // In Gen 1, a critical hit uses the base Attack or Special stat
  // For now, assume we've already factored this into `attack` parameter
  // or we recalculate it here using base stats
  if (isCritical) {
    // Critical hits in Gen 1 don't apply stat modifiers
    // If stat stages were applied, they're ignored for crits
    // (This is handled in the caller by passing the base stat)
    damage = Math.floor(damage * 1.0); // No multiplier, but base stat is used
  }

  // Step 3: Apply type effectiveness
  damage = Math.floor(damage * typeMultiplier);

  // Step 4: Apply random variance (0.85 to 1.00)
  damage = Math.floor(damage * randomFactor);

  // Step 5: Cap at 1 minimum
  if (damage < 1) {
    damage = 1;
  }

  return damage;
}

// Type Effectiveness Multiplier Determination
function getTypeMultiplier(
  moveType: string,
  defenderType1: string,
  defenderType2: string | null,
  isCritical: boolean
): number {
  // Critical hits ignore type multipliers and always use 1.0x
  // (Actually, crits in Gen 1 may ignore resistances but apply weaknesses — check Bulbapedia)
  // For now, assume crits ignore all multipliers (conservative interpretation)

  let multiplier = 1.0;

  // Check against primary type
  const mult1 = getSingleTypeMultiplier(moveType, defenderType1);
  multiplier *= mult1;

  // Check against secondary type (if exists)
  if (defenderType2) {
    const mult2 = getSingleTypeMultiplier(moveType, defenderType2);
    multiplier *= mult2;
  }

  // Clamp to 0, 0.5, 1, 2, or 2.5 range
  // (Gen 1 supports 0.5x and 2.5x in specific cases)
  return multiplier;
}

function getSingleTypeMultiplier(moveType: string, defenderType: string): number {
  // Immunities
  if ((moveType === "normal" && defenderType === "ghost") ||
      (moveType === "ghost" && defenderType === "normal")) {
    return 0;
  }

  // Super-effective (damage multiplier: 2.0x)
  const superEffective: Record<string, string[]> = {
    "fire": ["bug", "grass", "ice", "steel"],
    "water": ["fire", "ground", "rock"],
    "electric": ["water", "flying"],
    "grass": ["water", "ground", "rock"],
    "ice": ["flying", "ground", "grass"],
    "fighting": ["normal", "ice", "rock"],
    "poison": ["grass", "bug"],
    "ground": ["fire", "electric", "poison", "rock", "steel"],
    "flying": ["bug", "grass", "fighting"],
    "psychic": ["fighting", "poison"],
    "bug": ["grass", "psychic"],
    "rock": ["bug", "fire", "flying", "ice"],
    "ghost": ["ghost", "psychic"], // Note: Psychic immunity bug means this doesn't work
  };

  if (superEffective[moveType]?.includes(defenderType)) {
    return 2.0;
  }

  // Not very effective (damage multiplier: 0.5x)
  const notVeryEffective: Record<string, string[]> = {
    "fire": ["water", "ground", "rock"],
    "water": ["electric", "grass"],
    "electric": ["ground"],
    "grass": ["fire", "ice", "poison", "flying", "bug"],
    "ice": ["fire", "fighting", "rock", "steel"],
    "fighting": ["flying", "psychic"],
    "poison": ["ground", "psychic"],
    "ground": ["water", "grass", "ice"],
    "flying": ["electric", "ice", "rock"],
    "psychic": ["bug", "ghost"],
    "bug": ["fire", "flying", "rock"],
    "rock": ["water", "grass", "fighting", "ground", "steel"],
    "ghost": ["poison", "bug"],
  };

  if (notVeryEffective[moveType]?.includes(defenderType)) {
    return 0.5;
  }

  return 1.0;
}
```

### Key Differences from Later Gens

1. **Order of Operations**: Gen 1 applies damage modifiers in a specific order with integer division at each step. This differs significantly from Gen 3+.
2. **Integer Division Throughout**: Each step uses Math.floor(), creating specific damage thresholds and "jump" points where damage increases by 1.
3. **No Ability Modifiers**: No abilities to multiply damage.
4. **No Held Item Modifiers**: Items don't affect damage.
5. **Critical Hit Mechanics**: Crits use the base stat (ignoring stages) and may or may not ignore type effectiveness (needs verification).
6. **Random Factor**: Applied as a flat multiplier of 0.85 to 1.00 (inclusive on both ends in typical implementations, but verify Game Boy version).

### Damage Roll Range

The "random factor" from 0.85 to 1.00 is applied uniformly across all damage. This means a move that deals 100 damage has a roll range of 85-100 (with integer division applied after).

## 5. Critical Hits

Critical hit mechanics in Gen 1 are radically different from later generations.

### Critical Hit Rate Calculation

```typescript
interface CriticalHitCalculation {
  baseCritRate: number;  // Determined by attacker's base Speed
  focusEnergyUsed: boolean;  // If attacker used Focus Energy
}

function calculateCriticalHitChance(
  attackerBaseSpeed: number,
  focusEnergyUsed: boolean
): number {
  // Base crit rate is Speed / 512, clamped to 0-255 (for RNG purposes)
  let critRate = Math.floor(attackerBaseSpeed / 2);

  // Capped at 255 (meaning guaranteed crit in some cases with very high base Speed)
  critRate = Math.min(critRate, 255);

  // Apply Focus Energy (BUG: Divides by 4 instead of multiplying by 4)
  if (focusEnergyUsed) {
    // Intended: multiply by 4 (make crits more common)
    // Actual: divide by 4 (make crits much rarer, or fail to happen at all)
    critRate = Math.floor(critRate / 4);
  }

  // Return as probability 0.0 to 1.0
  return Math.min(critRate / 256, 1.0);
}

// Example crit rates:
// Base Speed 80: crit rate = 80/2 = 40/256 ≈ 15.6%
// Base Speed 100: crit rate = 100/2 = 50/256 ≈ 19.5%
// Base Speed 120: crit rate = 120/2 = 60/256 ≈ 23.4%
// Base Speed 20: crit rate = 20/2 = 10/256 ≈ 3.9%
```

### Focus Energy Bug

The move Focus Energy is supposed to increase critical hit rate by 4x. However, due to a programming bug, it actually divides the crit rate by 4, making it **much harder** to land critical hits.

```typescript
// Correct behavior (Gen 2+):
// critRate *= 4

// Gen 1 buggy behavior:
// critRate = Math.floor(critRate / 4)
// This is catastrophic for slow Pokémon and nearly breaks high-speed Pokémon
```

### Critical Hit Damage Recalculation

When a move lands as a critical hit in Gen 1:

```typescript
function applyCriticalHitDamage(

  attackerAttackStat: number,
  attackerAttackStages: number,
  attackerBaseAttack: number
): number {
  // Critical hits in Gen 1 ignore the attacker's stat stages
  // They use the base Attack (or Special) stat instead
  // This means if an attacker used Dragon Dance or Swords Dance, the boost is lost for crits

  // Recalculate damage using base stat
  // (This requires recalculating the entire damage formula from step 1 with base stats)

  // For implementation, re-run the damage formula but with:
  // attack = attackerBaseAttack (not the stage-modified Attack)

  return baseDamage; // (simplified; caller should recalculate)
}
```

Key points:
- **Speed determines crit chance**, not move properties or held items
- **Crit rate is Speed / 512** (then multiplied by 100 for percentage)
- **Focus Energy divides by 4** (bug that makes it useless or harmful)
- **Crits ignore stat stage boosts** and use the base stat
- **Crit rate maxes at ~100%** for base Speed 100+ (Alakazam at 120 base Speed has a ~23% crit rate without stacking)

## 6. Status Conditions

Gen 1 status conditions are simpler than later gens but have some unique quirks.

### Status Types and Effects

```typescript
// Gen 1 status conditions use the PrimaryStatus type from @pokemon-lib-ts/core
type Gen1PrimaryStatus = PrimaryStatus;

interface StatusState {
  condition: Gen1PrimaryStatus | null;
  counter: number;  // For Sleep (turns remaining) and Badly Poisoned (damage counter)
  turnsSinceApplication: number;  // For tracking when status was applied
}

function applyStatusEffect(
  condition: Gen1PrimaryStatus,
  target: Pokemon,
  applier: Pokemon
): StatusState {
  // Status application rules and effects
  return {
    condition,
    counter: 0,
    turnsSinceApplication: 0,
  };
}
```

### Burn

- **Effect**: Reduces Attack stat to 50% during damage calculation
- **Duration**: Permanent until cured (Antidote, Full Heal, Full Restore, etc.)
- **Interaction with Moves**: Moves like Reflect and Light Screen are not affected by Burn
- **Recovery**: Only via items or moves like Full Restore, not naturally

```typescript
function applyBurnEffect(attacker: Pokemon): number {
  // Reduce Attack to 50% for damage calculation
  if (attacker.status.condition === Gen1PrimaryStatus.Burn) {
    return Math.floor(attacker.stats.attack * 0.5);
  }
  return attacker.stats.attack;
}
```

### Freeze

- **Effect**: Pokémon cannot attack; fully paralyzed
- **Duration**: Permanent until cured
- **No Natural Thaw**: Unlike later gens, there is no "random 20% thaw chance" in Gen 1. A Frozen Pokémon stays frozen until hit with a Fire-type move (Burn or Sunny Day potentially, but verify) or cured by items.
- **Recovery**: Only via items or Fire-type moves (like Ember) hitting the Frozen Pokémon

### Paralysis

- **Effect**: Reduces Speed stat to 25% during turn order calculation
- **Duration**: Permanent until cured
- **Interaction with Move Accuracy**: No interaction in Gen 1; accuracy is not affected
- **Recovery**: Only via items (Antidote, Full Heal, Full Restore), not naturally

```typescript
function applyParalysisEffect(pokémon: Pokemon): number {
  // Reduce Speed to 25% for turn order calculation
  if (pokémon.status.condition === Gen1PrimaryStatus.Paralysis) {
    return Math.floor(pokémon.stats.speed * 0.25);
  }
  return pokémon.stats.speed;
}
```

### Sleep

- **Effect**: Pokémon cannot attack; fully inactive
- **Duration**: Determined by a counter (typically 1-7 turns in Gen 1)
- **Counter Mechanics**: When a Pokémon falls asleep, a counter is set (usually random 1-7). At the start of each turn while asleep, the counter decrements. When it hits 0, the Pokémon wakes up.
- **Counter Reset**: When a Sleeping Pokémon wakes up (counter reaches 0), the counter resets. If it's hit with a Wake-Up move before counter reaches 0, it wakes up immediately.
- **Recovery**: Naturally after counter expires, or via items/moves (Awaken, Full Restore)

```typescript
function handleSleepCounter(pokémon: Pokemon): void {
  if (pokémon.status.condition === Gen1PrimaryStatus.Sleep) {
    pokémon.status.counter -= 1;
    if (pokémon.status.counter <= 0) {
      pokémon.status.condition = Gen1PrimaryStatus.None;
      pokémon.status.counter = 0;
    }
  }
}
```

### Poison

- **Effect**: Regular Poison deals no damage in Gen 1 (unlike later gens with 1/8 HP per turn)
- **Duration**: Permanent until cured
- **Badly Poisoned**: A separate status condition applied by Toxic (if it exists in Gen 1). Badly Poisoned deals 1/8 HP damage per turn.
- **Switching Mechanic**: When a Poisoned Pokémon switches out, if it's Badly Poisoned, it resets to regular Poison

```typescript
function applyPoisonDamage(pokémon: Pokemon): void {
  // Regular Poison: no damage
  if (pokémon.status.condition === Gen1PrimaryStatus.Poison) {
    // No damage dealt
  }

  // Badly Poisoned: 1/8 HP damage
  if (pokémon.status.condition === Gen1PrimaryStatus.BadlyPoisoned) {
    const damage = Math.max(1, Math.floor(pokémon.maxHP / 8));
    pokémon.currentHP -= damage;
  }
}

function handlePoisonSwitch(pokémon: Pokemon): void {
  // When switching out, Badly Poisoned resets to regular Poison
  if (pokémon.status.condition === Gen1PrimaryStatus.BadlyPoisoned) {
    pokémon.status.condition = Gen1PrimaryStatus.Poison;
  }
}
```

### Stat Modifications

**Important**: In Gen 1, stat modifications (raised/lowered Attack, Defense, Speed, Special) are **transient** and **not** equivalent to status conditions. They wear off at the end of battle or when the Pokémon switches out (in some cases).

```typescript
interface StatModifiers {
  attack: number;     // -6 to +6 (each stage multiplies by 1.5x or divides by 1.5x)
  defense: number;
  speed: number;
  special: number;
  accuracy: number;   // -6 to +6
  evasion: number;    // -6 to +6
}

function applyStatStageModifier(
  baseStat: number,
  stage: number  // -6 to +6
): number {
  if (stage >= 1) {
    // Positive stages: multiply by (1 + stage * 0.5) or similar
    // Gen 1: (currentStat + boostAmount)
    // Actually, Gen 1 uses: statValue * (stage + 2) / 2
    // This means +1 is 1.5x, +2 is 2x, +3 is 2.5x, +4 is 3x, +5 is 3.5x, +6 is 4x
    return Math.floor(baseStat * (stage + 2) / 2);
  } else if (stage <= -1) {
    // Negative stages: divide
    // -1 is 2/3x, -2 is 1/2x, -3 is 2/5x, etc.
    return Math.floor(baseStat * 2 / (2 - stage));
  }
  return baseStat;
}
```

## 7. Move Mechanics

### Physical vs Special Determination

In Gen 1, a move's categorization as physical or special is determined **entirely by type**, not by any property of the move itself.

```typescript
interface MoveCategory {
  isPhysical: boolean;  // If true, uses Attack/Defense; if false, uses Special/Special
}

function getMoveCategoryByType(moveType: string): MoveCategory {
  const physicalTypes = [
    "normal", "flying", "fighting", "ground", "rock", "bug", "ghost"
  ];

  const specialTypes = [
    "fire", "water", "grass", "electric", "ice", "poison", "psychic"
  ];

  // Note: Some of these classifications seem arbitrary from a modern perspective
  // (e.g., Physical Flying moves use Attack, but Special Water moves use Special)

  return {
    isPhysical: physicalTypes.includes(moveType),
  };
}

// Example:
// Earthquake (Ground-type): Physical (uses Attack/Defense)
// Thunderbolt (Electric-type): Special (uses Special/Special)
// Hyper Beam (Normal-type): Physical (uses Attack/Defense) — this is a beam but still Physical!
// Hydro Pump (Water-type): Special (uses Special/Special)
```

### Move Properties by Category

Physical moves (Normal, Flying, Fighting, Ground, Rock, Bug, Ghost):
- Use Attack stat for damage
- Interact with Defense stat
- Can have accuracy checks
- Some physical moves have secondary effects (chance to paralyze, burn, etc.)

Special moves (Fire, Water, Grass, Electric, Ice, Poison, Psychic):
- Use Special stat for damage
- Interact with Special stat for defense
- Can have accuracy checks
- Some special moves have secondary effects

```typescript
    return true;

  const roll = Math.random() * 100;
  return roll < effectiveAccuracy;
}
```

## 8. Accuracy & Evasion

### Accuracy and Evasion Stages

Like other stats, Accuracy and Evasion can be modified by moves and abilities (though Abilities don't exist in Gen 1).

```typescript
function calculateAccuracyModifier(stage: number): number {
  // Accuracy has 9 stages: -4, -3, -2, -1, 0, +1, +2, +3, +4
  // But moves can boost beyond this
  // Accuracy multiplier = (stage + 3) / 3 if positive, 3 / (3 - stage) if negative
  // Actually, Gen 1 uses: [accuracy stages] multiplied into the formula
  // Let me use the correct Gen 1 formula

  // Gen 1 accuracy: stages are -6 to +6
  // Multiplier is calculated similar to stat stages but different formula
  // (3 + stage) / 3 for positive, 3 / (3 + stage) for negative
  // Verify this with Bulbapedia

  if (stage >= 0) {
    return (3 + stage) / 3;
  } else {
    return 3 / (3 + stage);
  }
}

function calculateEvasionModifier(stage: number): number {
  // Evasion works opposite to accuracy
  // Higher evasion makes you harder to hit
  // Applied as a divisor to accuracy

  if (stage >= 0) {
    return (3 + stage) / 3;
  } else {
    return 3 / (3 + stage);
  }
}

function resolveAccuracyCheck(
  moveBaseAccuracy: number,
  attackerAccuracyStage: number,
  defenderEvasionStage: number
): boolean {
  const accuracyMod = calculateAccuracyModifier(attackerAccuracyStage);
  const evasionMod = calculateEvasionModifier(defenderEvasionStage);

  let effectiveAccuracy = moveBaseAccuracy * accuracyMod / evasionMod;
  effectiveAccuracy = Math.min(100, Math.max(0, effectiveAccuracy));

  const roll = Math.floor(Math.random() * 256);
  return roll < (effectiveAccuracy / 100) * 256;
}
```

### The 1/256 Miss Glitch

This is one of the most infamous bugs in Pokémon Gen 1:

```typescript
// Move with 100% accuracy (like Hyper Beam, Swift, etc.)
function rollMoveAccuracy_BuggyGen1(
  move: Move,
  attacker: Pokemon,
  defender: Pokemon
): boolean {
  if (move.baseAccuracy === 100) {
    // Bug: moves with 100% accuracy are treated as having 255/256 accuracy
    // instead of being guaranteed to hit

    const roll = Math.floor(Math.random() * 256);

    // Intended: always return true
    // Actual: return roll < 255 (meaning 1/256 chance to miss)
    return roll < 255;
  }

  // Normal accuracy check
  const accuracyPercentage = move.baseAccuracy;
  const roll = Math.floor(Math.random() * 256);
  return roll < (accuracyPercentage / 100) * 256;
}

// This bug affects:
// - Hyper Beam (100% accuracy): 1/256 chance to miss despite being stated as never missing
// - Swift (100% accuracy): 1/256 chance to miss
// - Sky Attack (100% accuracy): 1/256 chance to miss
// - Many other 100% moves
//
// The bug is so prevalent that many competitive players account for it
// It's even a factor in some speedruns where runners reset if they hit the 1/256 miss
```

## 9. Turn Order

### Speed-Based Turn Order

Turn order in Gen 1 is determined purely by Speed stat (taking into account Paralysis reduction).

```typescript
function calculateActionOrder(
  pokemon1: Pokemon,
  pokemon2: Pokemon
): Pokemon[] {
  // Calculate effective Speed for each Pokémon
  const speed1 = getEffectiveSpeed(pokemon1);
  const speed2 = getEffectiveSpeed(pokemon2);

  if (speed1 > speed2) {
    return [pokemon1, pokemon2];
  } else if (speed2 > speed1) {
    return [pokemon2, pokemon1];
  } else {
    // Tie: random order (or, in some games, player always goes first)
    return Math.random() < 0.5 ? [pokemon1, pokemon2] : [pokemon2, pokemon1];
  }
}

function getEffectiveSpeed(pokémon: Pokemon): number {
  let speed = pokémon.stats.speed;

  // Apply stat stages
  speed = applyStatStageModifier(speed, pokémon.statStages.speed);

  // Apply Paralysis (reduces Speed to 25%)
  if (pokémon.status.condition === Gen1PrimaryStatus.Paralysis) {
    speed = Math.floor(speed * 0.25);
  }

  return speed;
}
```

### Priority System (Minimal in Gen 1)

Unlike later gens with priority brackets, Gen 1 has very few moves with priority:

```typescript
interface MoveWithPriority {
  name: string;
  priority: number;  // 0 for normal, +1 for quick moves, -1 for slower moves
}

const Gen1PriorityMoves: MoveWithPriority[] = [
  { name: "Quick Attack", priority: 1 },

function resolveTurnOrder(
  attacker1: Pokemon,
  attacker2: Pokemon,
  move1: Move,
  move2: Move
): [Pokemon, Move, Pokemon, Move] {
  // Get priority values (0 for most moves)
  const priority1 = move1.priority || 0;
  const priority2 = move2.priority || 0;

  if (priority1 > priority2) {
    return [attacker1, move1, attacker2, move2];
  } else if (priority2 > priority1) {
    return [attacker2, move2, attacker1, move1];
  } else {
    // Same priority: use Speed
    const [first, second] = calculateActionOrder(attacker1, attacker2);
    if (first === attacker1) {
      return [attacker1, move1, attacker2, move2];
    } else {
      return [attacker2, move2, attacker1, move1];
    }
  }
}
```

### Counter Move Mechanics

Counter is a move that acts with -1 priority and deals 2x the damage of the last move the opponent used.

```typescript
function resolveCounterMove(
  counterUser: Pokemon,
  opponent: Pokemon,
  opponentLastMove: Move,
  opponentLastDamage: number
): number {
  // Counter deals 2x the damage of the last move, but only if it was a Physical move
  // Counter is only effective against Physical moves
  if (opponentLastMove.isPhysical) {
    return opponentLastDamage * 2;
  } else {
    // Counter fails against Special moves
    return 0;
  }
}
```

## 10. End-of-Turn Effects

Gen 1 has a simpler set of end-of-turn effects compared to later gens.

### End-of-Turn Event Order

1. **Resolve move effects** (damage, secondary effects, stat changes)
2. **Apply status damage** (Burn, Poison, Badly Poisoned)
3. **Check for fainting** (if HP <= 0)
4. **Decrement timers** (Sleep counter, trapping counter)
5. **Decrement move counters** (recharge turns, etc.)

```typescript
function resolveEndOfTurn(
  pokemon1: Pokemon,
  pokemon2: Pokemon
): void {
  // Apply status damage
  applyStatusDamage(pokemon1);
  applyStatusDamage(pokemon2);

  // Decrement counters
  decrementSleepCounter(pokemon1);
  decrementSleepCounter(pokemon2);

  decrementTrappingCounter(pokemon1);
  decrementTrappingCounter(pokemon2);

  // Check for fainting (handled separately in the main battle loop)
}

function applyStatusDamage(pokémon: Pokemon): void {
  if (pokémon.status.condition === Gen1PrimaryStatus.BadlyPoisoned) {
    const damage = Math.max(1, Math.floor(pokémon.maxHP / 8));
    pokémon.currentHP = Math.max(0, pokémon.currentHP - damage);
  }
  // Regular Poison and Burn do no damage in Gen 1
  // Freeze and Sleep prevent action but don't deal damage
}
```

### No Weather, Terrain, or Hazards

Gen 1 has no weather system, terrain, or field effects like Stealth Rock, Spikes, or Reflect. Moves like Reflect and Light Screen can be used, but they only affect the attacker's team (in their generation of release).

### 6. Wrap and Trapping Moves Overpowered

Trapping moves are **extremely broken** in Gen 1:

- They prevent switching and lock the opponent in place
- They deal damage each turn (typically 1/16 HP)
- The opponent cannot take any action while trapped
- There's no way to escape (unlike later gens with moves like Shed Tail or abilities)
- Pokémon with access to Wrap (like Cloyster, Dragonite) could potentially single-handedly dominate battles

### 8. Psychic Type Dominance

In Gen 1, Psychic types are extremely overpowered:

- They only take super-effective damage from Bug and Ghost types
- Bug type attacks are rare and weak
- Ghost type attacks don't actually hit Psychic types (due to the immunity bug)
- Psychic moves are strong and have good coverage
- High-tier Psychic Pokémon like Alakazam, Gengar, and Espeon dominate competitive play

### Key Implementation Decisions

1. **Separate Stat Calculation**: The Gen1Ruleset should override the base stat calculation method entirely, as the formula differs from Gen 3+.

2. **Type-Based Move Categorization**: Instead of storing "Physical" or "Special" on each move, determine it at runtime based on type. This simplifies move definitions.

3. **Damage Formula Rewrite**: The Gen 1 damage formula is different enough that it warrants a completely separate implementation, not a parameterized version of the Gen 3+ formula.

4. **Critical Hit Recalculation**: When a critical hit lands, recalculate damage using base stats. This requires either:
   - A separate `calculateGen1DamageWithCrit()` method
   - Or passing a `useBaseStats: boolean` parameter to the damage calculation

5. **Status Condition Tracking**: Implement a StatusState struct that tracks:
   - The condition (Burn, Freeze, etc.)
   - A counter (for Sleep turns, Poison damage accumulation)
   - Whether the condition is "permanent" or temporary

6. **Stat Stage Application**: Stat stages should be stored as a separate `statStages` object on the Pokémon and applied via a multiplier function, not baked into the base stats.

7. **Minimal Priority System**: Since Gen 1 only has Quick Attack (+1) and Counter (-1), the priority system can be simplified. Don't create a full priority bracket system like Gen 5+.

8. **No Ability Checks**: Remove all ability-related logic from Gen 1 battles. Gen1Ruleset should never call any ability-related methods.

9. **No Held Items**: Ignore any held item effects in Gen 1 battles.

10. **Accurate Accuracy**: Implement the 1/256 miss bug for 100% accuracy moves. This is a known quirk that affects game balance.

### Testing Considerations

1. **Damage Calculation Edge Cases**: Test that the integer division at each step produces the correct minimum and maximum damage rolls.

2. **Critical Hit Rates**: Verify that base Speed 100 Pokémon have ~19.5% crit rate without stat boosts.

3. **Status Effects**: Ensure Freeze and Sleep never thaw/wake naturally, and that Burn reduces Attack to exactly 50%.

4. **Type Effectiveness**: Test the Ghost immunity bug and the 1/256 miss glitch.

5. **Turn Order**: Verify that Paralysis reduces Speed to 25% for turn order, and that Quick Attack always goes first.

6. **Stat Stages**: Confirm that stat stage modifiers are applied correctly (e.g., +1 Attack is 1.5x).
