# Data Pipeline Specification

> **Status: IMPLEMENTED** — This spec documents the data pipeline as actually built. The original design (PokeAPI local clone, modular architecture) was not implemented; the actual implementation uses @pkmn/dex + HTTP PokeAPI in a single file.

## Quick Start for AI Agents

**Entry point**: `tools/data-importer/src/import-gen.ts`

**Run a gen import**:
```bash
npx tsx tools/data-importer/src/import-gen.ts --gen=1
```

**Output**: JSON files written to `packages/gen1/data/`

**Primary source**: `@pkmn/dex` and `@pkmn/data` npm packages (Showdown data, battle-tested)

**Secondary source**: PokeAPI HTTP requests (for catch rate, base EXP, EV yield, exp group, height, and base friendship that Showdown doesn't track)

---

## 1. Overview

The data pipeline is a single-file TypeScript script (`tools/data-importer/src/import-gen.ts`) that transforms raw Pokemon data into per-generation JSON files. These JSON files are committed to the repository and ship with each `@pokemon-lib-ts/genN` npm package — consumers never need to run the importer themselves.

The script accepts a `--gen=N` CLI argument (1-9) and writes four JSON files into `packages/genN/data/`:

- `pokemon.json` — species data with base stats, types, abilities, learnsets, evolutions, and PokeAPI metadata
- `moves.json` — move data with type, power, accuracy, PP, flags, and effect descriptors
- `type-chart.json` — type effectiveness matrix for the generation
- `items.json` — held items and berries (empty array for Gen 1)

A `natures.json` stub (empty array) exists in each gen's data directory for interface consistency but is not generated by this script.

The script is run manually by maintainers when adding a new generation or updating data. It is not part of the automated build pipeline.

## 2. Data Sources

### 2.1 Primary: @pkmn/dex (Showdown Data)

The script uses `@pkmn/dex` and `@pkmn/data` npm packages to access Pokemon Showdown's battle data. These packages provide generation-scoped views of species, moves, items, types, learnsets, and abilities.

```typescript
import { Generations } from "@pkmn/data";
import { Dex } from "@pkmn/dex";

const gens = new Generations(Dex);
const gen = gens.get(GEN_NUM);  // Generation-scoped accessor
```

The `gen` object exposes:
- `gen.species` — iterable of all species in the generation
- `gen.moves` — iterable of all moves in the generation
- `gen.items` — iterable of all items in the generation
- `gen.types` — type effectiveness data
- `gen.learnsets.get(speciesId)` — async learnset lookup

**What @pkmn/dex provides**: base stats, types, abilities, move data (power/accuracy/PP/flags/secondary effects), item data, type effectiveness charts, learnsets, evolution chains, weight.

**What @pkmn/dex does NOT provide** (supplemented by PokeAPI): catch rate, base experience yield, EV yield, experience group/growth rate, height, base friendship.

### 2.2 Secondary: PokeAPI HTTP

For each Pokemon, the script makes two parallel HTTP requests to the live PokeAPI:

```
GET https://pokeapi.co/api/v2/pokemon-species/{dexNum}
  → capture_rate, base_happiness, growth_rate.name

GET https://pokeapi.co/api/v2/pokemon/{dexNum}
  → base_experience, height (decimetres), weight (hectograms), stats[].effort
```

A rate-limit delay of 80ms between species (approximately 12.5 requests/second) is applied to stay well under PokeAPI's rate limits.

**Field mappings from PokeAPI**:
- `capture_rate` → `catchRate`
- `base_experience` → `baseExp`
- `growth_rate.name` → `expGroup` (normalized via `GROWTH_RATE_MAP`)
- `stats[].effort` → `evYield` (keyed by internal stat name, only non-zero entries)
- `height / 10` (dm → m) → `dimensions.height`
- `base_happiness` → `baseFriendship`

Weight comes from `@pkmn/dex` (`species.weightkg`), with PokeAPI weight as a fallback.

**PokeAPI stat name mapping** (`STAT_NAME_MAP`):

| PokeAPI name      | Internal name |
|-------------------|---------------|
| `hp`              | `hp`          |
| `attack`          | `attack`      |
| `defense`         | `defense`     |
| `special-attack`  | `spAttack`    |
| `special-defense` | `spDefense`   |
| `speed`           | `speed`       |

**Growth rate normalization** (`GROWTH_RATE_MAP`):

| PokeAPI value | Normalized value |
|---------------|-----------------|
| `medium-fast` | `medium-fast`   |
| `medium-slow` | `medium-slow`   |
| `medium`      | `medium-fast`   |
| `fast`        | `fast`          |
| `slow`        | `slow`          |
| `erratic`     | `erratic`       |
| `fluctuating` | `fluctuating`   |

## 3. Generated Output Files

For each generation, the importer produces JSON files in `packages/genN/data/`. These files are committed and ship with the npm package.

### 3.1 pokemon.json

Array of species objects. One entry per base form (no alternate formes). Ordered by National Dex number.

National Dex ranges per generation:

| Gen | Range   | Count |
|-----|---------|-------|
| 1   | 1–151   | 151   |
| 2   | 1–251   | 251   |
| 3   | 1–386   | 386   |
| 4   | 1–493   | 493   |
| 5   | 1–649   | 649   |
| 6   | 1–721   | 721   |
| 7   | 1–809   | 809   |
| 8   | 1–905   | 905   |
| 9   | 1–1025  | 1025  |

Each entry shape:
```typescript
{
  id: number,              // National Dex number
  name: string,            // Showdown ID (lowercase, no spaces, e.g. "bulbasaur")
  displayName: string,     // Display name (e.g. "Bulbasaur")
  types: string[],         // Lowercase type names (e.g. ["grass", "poison"])
  baseStats: {
    hp: number,
    attack: number,
    defense: number,
    spAttack: number,
    spDefense: number,
    speed: number,
  },
  abilities: {
    normal: string[],      // Kebab-case ability IDs; empty array for Gen 1-2
    hidden: string | null, // Null for Gen 1-2
  },
  genderRatio: number,     // % male (0-100); -1 for genderless
  catchRate: number,       // From PokeAPI
  baseExp: number,         // From PokeAPI
  expGroup: string,        // Growth rate (e.g. "medium-fast")
  evYield: Record<string, number>,  // Only non-zero stats
  eggGroups: string[],     // Lowercase
  learnset: {
    levelUp: Array<{ level: number, move: string }>,
    tm: string[],
    egg: string[],
    tutor: string[],
  },
  evolution: {
    from: EvolutionLink | null,
    to: EvolutionLink[],
  },
  dimensions: {
    height: number,        // Metres (from PokeAPI)
    weight: number,        // Kg (from @pkmn/dex, fallback PokeAPI)
  },
  spriteKey: string,       // Same as name
  baseFriendship: number,  // From PokeAPI
  generation: number,      // Generation the species was introduced
  isLegendary: boolean,
  isMythical: boolean,
}
```

### 3.2 moves.json

Array of move objects. Sorted alphabetically by kebab ID. Max/Z-moves and nonstandard moves are excluded.

Each entry shape:
```typescript
{
  id: string,              // Kebab-case (e.g. "thunder-wave")
  displayName: string,     // Display name (e.g. "Thunder Wave")
  type: string,            // Lowercase type name
  category: string,        // "physical" | "special" | "status"
  power: number | null,    // null for status moves
  accuracy: number | null, // null for never-miss moves
  pp: number,
  priority: number,
  target: string,          // See target mapping below
  flags: {
    contact: boolean,
    sound: boolean,
    bullet: boolean,
    pulse: boolean,
    punch: boolean,
    bite: boolean,
    wind: boolean,
    slicing: boolean,
    powder: boolean,
    protect: boolean,
    mirror: boolean,
    snatch: boolean,
    gravity: boolean,
    defrost: boolean,
    recharge: boolean,
    charge: boolean,
    bypassSubstitute: boolean,
  },
  effect: object | null,   // Effect descriptor (see Section 5.2)
  description: string,
  generation: number,
}
```

### 3.3 type-chart.json

Nested object mapping `attackType → defenseType → multiplier`. Multiplier values are `0`, `0.5`, `1`, or `2`.

Type lists per generation:
- **Gen 1**: 15 types (no Dark, Steel, Fairy)
- **Gen 2–5**: 17 types (adds Dark, Steel)
- **Gen 6–9**: 18 types (adds Fairy)

### 3.4 items.json

Array of item objects sorted alphabetically. **Empty array for Gen 1** (no items in Gen 1).

Each entry shape:
```typescript
{
  id: string,              // Kebab-case
  displayName: string,
  description: string,
  category: string,        // "held-item" | "berry" | "pokeball" | "evolution-item" | "mail"
  pocket: string,          // "items" | "berries" | "pokeballs"
  price: number,           // Always 0 (@pkmn/dex doesn't provide prices)
  battleUsable: boolean,
  fieldUsable: boolean,    // Always false
  generation: number,
  spriteKey: string,
  holdEffect?: object,     // Present only if the item has a hold effect
}
```

## 4. CLI Usage

```bash
npx tsx tools/data-importer/src/import-gen.ts --gen=N
```

`N` must be an integer from 1 to 9.

There is only one CLI flag. There is no `--all`, `--validate`, or `--dry-run` flag. Running without `--gen=N` prints a usage message and exits with code 1.

The script writes to `packages/genN/data/` relative to the monorepo root (resolved via `import.meta.dirname`). The output directory is created automatically if it does not exist.

Progress is logged to stdout:

## 5. Data Transformation

### 5.1 Name and ID Normalization

**Pokemon names**: The internal `name` field uses the Showdown species ID (lowercase, no spaces or special characters). The `displayName` field uses the human-readable Showdown name.

**Move IDs**: Converted from the Showdown display name using `moveIdToKebab()`:
- Apostrophes (`'`, `'`) are stripped
- Non-alphanumeric characters become hyphens
- Leading/trailing hyphens removed
- Lowercased

**Ability and item IDs**: Converted via `toKebab()`:
- Strips apostrophes
- Inserts hyphens at camelCase boundaries
- Whitespace becomes hyphens
- Lowercased

### 5.2 Move Categories (Gen 1–3)

In Gen 1–3, physical/special split is determined by the move's type, not per-move. The `getTypeBasedCategory()` function overrides Showdown's per-move category for Gen 1–3:

**Physical types**: normal, fighting, flying, ground, rock, bug, ghost, poison, steel
**Special types**: fire, water, electric, grass, ice, psychic, dragon, dark

Status category is passed through unchanged regardless of generation.

Gen 4+ uses Showdown's per-move category directly.

### 5.3 Move Target Mapping

Showdown target strings are mapped to internal target names:

| Showdown          | Internal            |
|-------------------|---------------------|
| `normal`          | `adjacent-foe`      |
| `allAdjacentFoes` | `all-adjacent-foes` |
| `allAdjacent`     | `all-adjacent`      |
| `self`            | `self`              |
| `allySide`        | `user-field`        |
| `allyTeam`        | `user-and-allies`   |
| `foeSide`         | `foe-field`         |
| `all`             | `entire-field`      |
| `randomNormal`    | `random-foe`        |
| `any`             | `any`               |
| `scripted`        | `adjacent-foe`      |
| `adjacentAlly`    | `adjacent-ally`     |
| `adjacentAllyOrSelf` | `self`           |
| `adjacentFoe`     | `adjacent-foe`      |

### 5.4 Move Effect Descriptors

The `buildMoveEffect()` function converts Showdown move data into structured effect objects. The shape varies by move type:

| Effect type | Shape |
|-------------|-------|
| `fixed-damage` | `{ type, damage: number }` |
| `level-damage` | `{ type }` |
| `ohko` | `{ type }` |
| `multi-hit` | `{ type, min: number, max: number }` |
| `two-turn` | `{ type, firstTurn: string }` |
| `heal` | `{ type, amount: number }` (fraction of max HP) |
| `drain` | `{ type, amount: number }` (fraction of damage dealt) |
| `recoil` | `{ type, amount: number }` (fraction of damage dealt) |
| `weather` | `{ type, weather: string, turns: 5 }` |
| `status-guaranteed` | `{ type, status: string }` |
| `status-chance` | `{ type, status: string, chance: number }` |
| `volatile-status` | `{ type, status: string, chance: number }` |
| `stat-change` | `{ type, changes: Array<{stat, stages}>, target: "self"\|"foe", chance: number }` |
| `entry-hazard` | `{ type, hazard: string }` |
| `screen` | `{ type, screen: string, turns: 5 }` |
| `protect` | `{ type: "protect", variant: "standard" }` |
| `switch-out` | `{ type, who: "self" }` |
| `custom` | `{ type: "custom", handler: string }` |
| `multi` | `{ type: "multi", effects: object[] }` |

Status abbreviations are mapped: `brn`→`burn`, `par`→`paralysis`, `psn`→`poison`, `tox`→`badly-poisoned`, `slp`→`sleep`, `frz`→`freeze`.

Moves with multiple effects (e.g., drain + recoil) are wrapped in a `multi` effect. Moves with no recognized effect return `null`.

### 5.5 Learnset Parsing

Learnsets are fetched from Showdown via `gen.learnsets.get(speciesId)`. Each move source string has the format `{gen}{method}{data}` where method is one of:

| Code | Meaning | Data |
|------|---------|------|
| `L`  | Level-up | Level number |
| `M`  | TM/HM   | (none used) |
| `E`  | Egg move | (none) |
| `T`  | Move tutor | (none) |
| `S`  | Special/event | (skipped) |

Only sources matching the target generation prefix are included. The `levelUp` array is sorted by level then alphabetically, with duplicates removed.

### 5.6 Evolution Links

Evolution data comes entirely from Showdown (`species.prevo`, `species.evos`, `species.evoType`, `species.evoLevel`, `species.evoItem`, `species.evoMove`, `species.evoCondition`).

`mapEvoType()` normalizes Showdown's evolution type strings:

| Showdown evoType | Internal method |
|------------------|-----------------|
| (none/levelExtra) | `level-up` or `special` |
| `trade`          | `trade`         |
| `useItem`        | `use-item`      |
| `levelFriendship` + day condition | `friendship-day` |
| `levelFriendship` + night condition | `friendship-night` |
| `levelFriendship` (other) | `friendship` |
| `levelMove`      | `level-up`      |
| `levelHold`      | `level-up`      |
| other            | `special`       |

The standard friendship threshold (220) is set on all friendship-method evolutions.

### 5.7 Abilities

For Gen 1–2, abilities are set to `{ normal: [], hidden: null }` regardless of what Showdown reports.

For Gen 3+, abilities are read from the `species.abilities` object:
- Slots `"0"` and `"1"` → `normal[]` (kebab-cased)
- Slot `"H"` → `hidden` (kebab-cased, or null if absent)
- Slot `"S"` is ignored

### 5.8 Legendary/Mythical Detection

Tags from `species.tags` are checked:
- `"Restricted Legendary"` or `"Sub-Legendary"` → `isLegendary: true`
- `"Mythical"` → `isMythical: true`

## 6. Generation-Specific Considerations

### Gen 1

- **No items**: `buildItemsData()` returns an empty array immediately for Gen 1.
- **No abilities**: Ability slots are hardcoded to `{ normal: [], hidden: null }`.
- **15-type chart**: No Dark, Steel, or Fairy types.
- **Type-based physical/special**: All moves use type-based category assignment.
- **`???` type**: Showdown uses `???` for Curse in some gens; mapped to `ghost`.
- **Dex range**: 1–151.

### Gen 2

- **Items included**: The first gen with a non-empty `items.json`.
- **No abilities**: Ability slots are still hardcoded to empty.
- **17-type chart**: Adds Dark and Steel.
- **Type-based physical/special**: Same as Gen 1.
- **Dex range**: 1–251 (cumulative — includes all Gen 1 Pokemon).
- **Berry system**: Gen 2 uses an older berry system with named berries (Berry, Gold Berry, etc.) rather than the Gen 3+ Sitrus/Oran system.

### Gen 3+

- **Abilities**: Populated from Showdown data.
- **Per-move category**: Physical/special determined per move, not by type.
- **Fairy type**: Added in Gen 6.

### Type Chart Generation

The script hardcodes which types exist per generation (`GEN_TYPES`) and queries Showdown for effectiveness values. If a type is not found in Showdown's data for the target generation, neutral (1.0) effectiveness is used for all matchups involving that type, with a console warning.
